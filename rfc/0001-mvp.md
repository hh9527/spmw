# RFC 0001: MVP 对象存储和激活模型

## 状态

已完成。

完成说明：本文按 MVP 落地后的行为校准，记录已经稳定下来的核心模型。
更细的操作用法以 `README.md` 和当前脚本实现为准。

## 目标

MVP 内部把 setup 拆成四个明确阶段：

- `update`：生成下一次安装计划的输入。
- `install`：物化不可变对象，并生成具体 plan。CLI 默认随后执行内部 activate。
- `activate`：根据已完成的 plan 应用全局资源。
- `prune`：删除过期的全局资源，并在显式请求时删除不再使用的共享对象。

这个设计把全局状态变更和对象安装分开。对象一旦 ready 就不可变。全局资源删除只由 `prune` 执行。
CLI 只暴露 `update`、`install` 和 `prune`。其中 `install` 默认会在生成 plan 后继续执行
内部 activate；`install -Prepare` 只物化对象并生成 plan。

## 对象存储

对象放在 `.spmw/object` 下。

```text
.spmw/
  object/
    pkgs/
      <pkg-name>.<path-hash>/
        .READY
    fonts/
      <font-base>.<content-hash>.<ext>
    dl/
      <hash-of-src>-<file>
  state/
    next-plan.json
    plan/
      <plan-hash>.json
    lock.json
```

repo 视作一种 package。工具自身也可以作为一个 package 被安装和更新。

本项目保持工具属性，不内置具体配置。具体配置由 repo package 提供。

配置 repo 的 package key 固定为 `main`。`main` 和 `spmw` 是有特殊含义的
package 名称，但在配置模型上仍然都放在 `packages` 下。其他 package 都由
`main` 的 `config.spmw.json` 声明。

虚拟根 `bin` 表示用户可执行入口目录。虚拟根 `apps` 表示开始菜单中的固定
`SPMW` 子菜单，里面放 Windows 快捷方式。

package 对象按目录粒度共享：

```text
object/pkgs/<pkg-name>.<path-hash>/
```

package 的路径由 resolved variables 中的 `path` 变量决定。MVP 中 `path` 的形式是：

```text
pkgs/<pkg-name>.<variable-digest 前 16 个 hex 字符>
```

`variable-digest` 本身仍然是完整 SHA-256。package 对象只有在 `.READY` 存在时才算完成。

font 对象按文件粒度共享：

```text
object/fonts/<font-base>.<content-hash>.<ext>
```

下载文件按展开后的 `src` 和 `file` 共同派生的文件名共享：

```text
object/dl/<hash-of-src>-<file>
```

`file` 是下载对象的逻辑文件名，不是完整路径。真实路径始终带有 `src` hash
前缀，以避免不同来源的同名文件冲突。

如果 install action 没有指定 `file`，则 `file = basename(src)`。这表示
`src` 本身是稳定的不可变 URL。

如果 install action 指定了 `file`，则使用展开后的 `file` 作为稳定的逻辑文件名。
这用于 URL 可变、但内容由变量或校验规则固定的下载。例如 nightly URL 可以指定：

```json
{
  "file": "WezTerm-<sha256>.zip",
  "src": "https://example.com/WezTerm-nightly.zip",
  "verify": {
    "sha256": "<sha256>"
  }
}
```

不再提供按时间戳保存可变下载的通用机制。只要 `src` 不是稳定的不可变 URL，
install action 就必须显式指定 `file`，把下载对象的逻辑文件名固定下来。

package readiness metadata 中应记录下载文件的真实内容 hash。URL hash 只能标识来源，不能标识字节内容。

## 配置仓同步

配置部分托管在独立 repo 中。MVP 中配置 repo 按普通 package 的规则同步和物化，
只是它的 key 固定为 `main`。

生产入口由配置仓提供的 `bootstrap.ps1` 负责。bootstrap 先下载一份临时
`spmw` 启动器，再下载配置仓的 `config.spmw.json`，过滤出自举必需的
`packages.main`、`packages.spmw` 和正式 CLI link，然后执行：

```powershell
bootstrap/spmw-cli.ps1 update -Bootstrap <filtered-config>
bootstrap/spmw-cli.ps1 install
~/.local/bin/spmw-cli.ps1 update
~/.local/bin/spmw-cli.ps1 install
```

自举完成后，不再依赖单独的 `~/.config.spmw.json`。后续 `update` 从
`state/next-plan.json` 中的 `packages.main.variables.path` 推导当前配置入口：

```text
object:<main.path>/config.spmw.json
```

`main` 指示远端配置 repo 的同步来源。一个典型配置形态是：

```json
{
  "packages": {
    "main": {
      "variables": [
        {
          "commit": {
            "src": "https://github.com/OWNER/REPO/commits/main.atom",
            "ty": "CommitFromGithubAtom"
          }
        }
      ],
      "install": [
        {
          "action": "InstallMainConfig",
          "file": "windows-config-<commit>.tar.gz",
          "src": "https://github.com/OWNER/REPO/archive/<commit>.tar.gz"
        }
      ]
    }
  }
}
```

`update` 先按普通 package 规则解析 `main` variables：

```json
{
  "manifest-digest": "sha-of-main-manifest",
  "commit": "xxx",
  "variable-digest": "sha-of-previous-variables",
  "path": "pkgs/main.<short-hash>"
}
```

这组 `main` variables 会写入 `state/next-plan.json`，作为后续 `install`
定位配置入口和复现 plan 输入的一部分。

变量生成规则是确定的：

- 第一个变量是内置的 `manifest-digest`，来自 package 配置本身的 hash。
- 中间变量来自 package 配置中的本地计算或远程获取。
- 最后一个变量是内置的 `variable-digest`，它是之前所有变量的稳定摘要。
- `path` 变量在 `variable-digest` 之后生成，用来决定 package object 路径。
- 下载地址、安装动作等 package 配置字段不是 variables；它们通过 package 配置和
  variables 展开得到。

远程文本变量使用统一 resolver。`{ "src": "..." }` 等价于
`{ "src": "...", "ty": "Text" }`：下载 `src`，读取文本并 trim 后作为变量值。
参与 `manifest-digest` 的 package definition 使用规范化形式，
也就是会把缺省的 `ty` 补成 `"Text"`，避免简写和显式写法产生不同摘要。

`update` 使用 `main` 的 install action 展开出的 URL 下载 tarball，并用
`verify` 中声明的规则校验真实内容。下载到的 tarball 按普通下载对象处理，解包后物化为：

```text
object/pkgs/main.<path-hash>/
```

`main` package ready 后，`update` 从其中读取远端配置：

```text
config.spmw.json
```

然后把入口配置和远端 `config.spmw.json` 合并，再按合并后的配置继续正常处理。
MVP 的叠加规则是：`packages`、`links` 和 `shortcuts` 按 key 合并。正常更新时
远端同名 key 覆盖入口配置；`-Bootstrap` 阶段只使用传入的过滤配置。
合并后的 `packages.main` 也会进入普通 package 安装流程；由于对象已在同步配置时
物化，第二次安装应是幂等 no-op，不产生额外副作用。

## 原子性

对象和 state 文件在写入过程中使用 `.tmp.` 名称。非 `.tmp.` 名称表示文件或对象已经提交。

示例：

```text
object/pkgs/.tmp.ripgrep.<hash>/
object/fonts/.tmp.Cascadia.<hash>.ttf
object/dl/.tmp.<hash-of-src>-<file>
state/.tmp.next-plan.json
state/.tmp.lock.json
state/plan/.tmp.<plan-hash>.json
```

对于 package 对象，`.READY` 在临时 package 目录内写入，然后该目录才会 rename 到最终名称。`.READY` 和 package 目录同生命周期。

临时文件和目录在获得进程锁后自动清理。它们不受 `prune` 选项控制。

## State 文件

### `state/next-plan.json`

`update` 写入 `state/next-plan.json`。它不是 active state，也不保存完整
package definition、links 或 shortcuts。MVP 中它只保存下一次 install 需要的
resolved variables 快照：

```json
{
  "schema": 1,
  "packages": {
    "main": {
      "variables": {
        "manifest-digest": "...",
        "commit": "...",
        "variable-digest": "...",
        "path": "pkgs/main.<hash>"
      }
    },
    "spmw": {
      "variables": {
        "manifest-digest": "...",
        "version": "...",
        "variable-digest": "...",
        "path": "pkgs/spmw.<hash>"
      }
    }
  }
}
```

`install` 根据其中的 `packages.main.variables.path` 重新读取
`object:<main.path>/config.spmw.json`，再使用 next-plan 中已经解析好的
variables 执行安装。

### `state/plan/<plan-hash>.json`

`install` 在所有需要的 package 对象 ready 后写入 `state/plan/<plan-hash>.json`。

plan hash 由完整 plan 内容决定。plan 包含完整安装结果，以及 activation 必须应用的 resources。

示例结构：

```json
{
  "schema": 1,
  "id": "<plan-hash>",
  "plan": {
    "packages": {
      "main": {
        "object": "object:pkgs/main.<hash>",
        "variables": {
          "manifest-digest": "...",
          "commit": "...",
          "variable-digest": "...",
          "path": "pkgs/main.<hash>"
        }
      },
      "WezTerm": {
        "object": "object:pkgs/WezTerm.<hash>",
        "variables": {
          "manifest-digest": "...",
          "sha256": "...",
          "variable-digest": "...",
          "version": "nightly-...",
          "path": "pkgs/WezTerm.<hash>"
        }
      },
      "TermSCP": {
        "object": "object:pkgs/TermSCP.<hash>",
        "variables": {
          "manifest-digest": "...",
          "variable-digest": "...",
          "version": "1.1.1",
          "path": "pkgs/TermSCP.<hash>"
        }
      }
    }
  },
  "resources": {
    "links": [
      {
        "key": "link:bin/wezterm.exe",
        "target": "object:pkgs/WezTerm.<hash>/wezterm.exe"
      }
    ],
    "shortcuts": [
      {
        "key": "shortcut:apps/WezTerm",
        "program": "object:pkgs/WezTerm.<hash>/wezterm.exe",
        "cwd": "object:pkgs/WezTerm.<hash>"
      }
    ],
    "regs": []
  }
}
```

`activate` 消费这个文件。它不解析变量，也不安装 package。

### `state/lock.json`

`state/lock.json` 记录当前已激活的 plan，以及本工具已经接管的全局资源 key。

示例结构：

```json
{
  "schema": 1,
  "plan": "<current-plan-hash>",
  "refs": [
    "link:bin/rg.exe",
    "reg:HKCU/Software/Microsoft/Windows NT/CurrentVersion/Fonts/Example (TrueType)"
  ]
}
```

`plan` 是当前已经 activate 的 plan。

`refs` 是本工具已经接管过的全局 resource keys。它只保存 key，不保存 value。`activate` 期间它只追加；只有 `prune` 可以删除 key，并且必须在对应真实全局资源删除成功后才能删除 key。

## 命令

### `update`

`update` 准备下一次 plan 的输入。

它会：

- `-Bootstrap <path>` 时读取传入的过滤配置；普通运行时从当前
  `next-plan.json` 的 `main.path` 推导入口配置
- 解析入口配置中 `packages.main` 的 resolved variables
- 同步 `main` 配置 repo，并按 `verify` 声明校验内容
- 将配置 repo 物化为 `object/pkgs/main.<hash>`
- 读取 `main/config.spmw.json`，并和入口配置合并
- 运行合并后配置定义的变量更新逻辑
- 记录各 package 的 resolved variables
- 原子写入 `state/next-plan.json`

它不安装 package，不应用 link，不写 registry，也不更新 `state/lock.json`。

### `install`

`install` 消费 `state/next-plan.json`。

它会：

- 安装所有需要的 package 对象到 `object/pkgs`
- 在提交 package 目录前写入每个 package 的 `.READY`
- 在 `object/fonts` 下创建或复用共享 font 文件
- 在 `object/dl` 下创建或复用共享下载文件，并按 `verify` 校验内容
- 写入 package readiness metadata，包括 downloads、verify 结果、font refs 和 package 产生的 resources
- 根据 `main/config.spmw.json` 生成 links 和 shortcuts
- 将每个 package 的 object 引用、resolved variables 和 resources 写入 plan
- 写入 `state/plan/<plan-hash>.json`

默认情况下，`install` 在 plan 写入成功后继续执行 activate，应用全局 link 和 registry，
并更新 `state/lock.json`。

`install -Prepare` 只执行上述对象安装和 plan 生成步骤，不修改全局 link 或 registry，
也不更新 `state/lock.json`。

### 内部 activate 阶段

activate 是内部阶段，不作为 CLI 命令暴露。它消费一个已完成的 plan。

它会：

- 读取 `state/plan/<plan-hash>.json`
- 根据 `resources.links`、`resources.shortcuts` 和 `resources.regs`
  upsert 全局 link、Windows 快捷方式和 registry value
- 将成功应用的 resource keys 追加到 `state/lock.json.refs`
- 将 `state/lock.json.plan` 更新为已 activate 的 plan

它不删除过期的全局资源。过期资源由 `prune` 处理。

### `prune`

`prune` 是唯一会删除过期全局资源的命令。

默认情况下，它清理：

- `state/lock.json.refs` 中存在、但当前 plan 不再需要的全局 link 和 registry value
- 当前 plan 不再需要的过期 plan 文件和 repo package 对象

对于全局资源，`prune` 必须先删除真实 link 或 registry value。只有真实删除成功后，才能从 `state/lock.json.refs` 中移除对应 key。

共享对象清理由显式选项控制：

- `-Pkgs`：删除保留 plan 不再可达的 package 对象。
- `-Fonts`：删除保留 package readiness metadata 不再引用的 font 文件。
- `-Cache`：删除保留 package readiness metadata 不再引用的下载缓存文件。

完整共享对象清理可以显式组合这些选项：

```powershell
.\spmw-cli.ps1 prune -Pkgs -Fonts -Cache
```

## 安全规则

- `install -Prepare` 只创建或复用不可变对象，并生成 plan。
- `install` 默认在 prepare 成功后执行内部 activate 阶段。
- 内部 activate 阶段只应用或更新全局资源，并记录 owned keys。
- 内部 activate 阶段永远不删除全局资源。
- `prune` 保守删除全局资源，并且只在真实删除成功后更新 `refs`。
- 缺少 `.READY` 表示 package 对象不完整，不能使用。
- `.tmp.` 条目是不完整对象，可以在进程锁保护下自动移除。
- registry font 发生 apply 或 delete 后，应广播 Windows font change notification。
